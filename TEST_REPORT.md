# V18.4.3 富文本字号问题修复报告

## 问题一：选中文字较多时调整字号偶发无响应

### 根因A：原选区在操作下拉框时被清除

字号控件是原生 `<select>`。用户打开下拉框时，浏览器会把焦点从正文移动到字号控件，并临时折叠正文选区。

原来的 `selectionchange` 处理随后调用 `hideRichSelectionToolbar()`，同时把 `savedRichRange` 清空。等 `change` 事件真正应用字号时，已经找不到原选区，因此看起来“没有反应”。

### 根因B：大选区被放进单个外层 span

旧逻辑把整个 `Range.extractContents()` 结果放入一个 `<span>`。当选择跨越多行、块级元素或多个已有样式节点时，这会形成不稳定或无效的 HTML 层次，浏览器可能重排节点，使部分字号没有生效。

### 修复

- 工具条正在操作或拥有焦点时，不清除 `savedRichRange`；
- 字号控件变更后恢复原编辑器 Range；
- 提取选区后遍历其中的所有文本节点；
- 为每个文本片段增加字号 span，保留内部颜色、粗体和高亮；
- 将完整插入内容重新设为选中状态。

## 问题二：新文本框显示上一次字号

### 根因

新自由文字的数据实际仍是默认 `22px`，但字号 `<select>` 没有在打开新编辑器时同步，只保留了上一次操作的界面值。

因此出现：

```text
工具条显示上次字号
实际文字仍是22px
```

### 修复

- 每次打开编辑器都读取当前编辑对象的计算字号；
- 选区变化时重新统计所选文字的计算字号；
- 新自由文字显示22；
- 节点正文显示16；
- 多种字号混选时显示“混合字号”；
- 不把上一次字号作为新文本框的数据默认值。
