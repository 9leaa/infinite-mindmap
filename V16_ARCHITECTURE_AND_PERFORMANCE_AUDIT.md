# 无限画布思维导图 V16：稳定架构与性能审计

## 一、为什么不再继续修补 V15

V15 的根本问题不是单一阈值或按钮，而是节点元数据、大型附件、全部手写坐标和 PDF 批注共用保存、撤销与渲染链路。一次很小的操作，也可能触发整个笔记库的复制、序列化或重绘。

V16 将这些职责拆开，不再把所有数据作为一个大对象反复保存。

## 二、V16 数据架构

| IndexedDB Store | 保存内容 | 写入时机 |
|---|---|---|
| `library` | 节点、关系、自由对象、主题和工具设置 | 结构变化或空闲期 |
| `binary-assets` | PDF、DOC、PPT 原始 Blob | 文件导入时一次 |
| `ink-strokes` | 每一条画布笔迹 | 一笔完成、擦除或撤销时 |
| `pdf-annotations` | 每一条 PDF 页面批注 | 一条批注完成或删除时 |

普通自动保存不会再复制 PDF、Office、全部历史笔迹或全部 PDF 批注。

## 三、手写路径

### 落笔过程中

- 只向当前 Canvas 增量绘制新线段。
- 不保存整个笔记库。
- 不生成完整撤销快照。
- 不执行整笔后台简化任务。
- 不重建节点列表。

### 抬笔时

只执行：

1. 完成当前笔迹对象。
2. 计算这一笔的边界框。
3. 将这一条记录写入 `ink-strokes`。
4. 将一条“新增笔迹”命令加入轻量撤销历史。

不会遍历全部旧笔迹。

### 撤销

手写使用笔画级命令：

- 新增一笔；
- 删除一笔；
- 像素橡皮产生的笔迹分割差异。

不再将整张画布转换成完整 JSON 快照。

## 四、Pencil 与手指输入

V16 只使用 Pointer Events：

- Apple Pencil：`pointerType = "pen"`；
- 手指：`pointerType = "touch"`；
- 鼠标：`pointerType = "mouse"`。

不再同时监听 Pointer Events 和 Touch Events，因此不会把同一次接触处理两遍。

在钢笔、水彩、激光和橡皮模式中：

- Pencil 执行当前绘图工具；
- Pencil 抬起后，手指立即可以平移；
- 手指平移使用约 4px 启动阈值；
- Pencil 仍落在屏幕上时，不允许改变画布坐标系，并显示明确提示。

## 五、画布渲染

### 平移

拖动画布时：

1. 节点层使用 CSS Transform。
2. 已有手写 Canvas 使用 GPU `translate3d` 临时移动。
3. 拖动过程中不遍历全部历史笔迹。
4. 手指抬起后只完整重绘一次。

该机制用于绘图模式单指平移、选择模式空白平移和触控板滚动平移。

### 笔迹视口裁剪

每条笔迹保存边界框。完整重绘前先判断整条笔迹是否与视口相交。屏幕外笔迹不会继续遍历所有坐标点。

### 节点与自由对象拖动

移动过程中只修改受影响 DOM 元素的 `left/top`：

- 不再每个 Pointer Move 重建全部节点；
- 不再每个 Pointer Move 重建全部自由对象；
- 关系线刷新限制在约 30 FPS；
- Pointer Up 后只进行一次完整同步。

## 六、PDF 双模式

### 快速阅读

导入 PDF 时：

- 只将原始 File Blob 写入 `binary-assets`；
- 不加载 PDF.js；
- 不解析页数；
- 不创建页面 Canvas。

点击“打开阅读”后，PDF 先交给浏览器原生 PDF 引擎，以 Blob URL 立即显示。

### 批注模式

只有用户主动点击“批注模式”、钢笔、荧光笔或橡皮时才：

1. 加载本地 `vendor/pdf.min.mjs`；
2. 启动 PDF Worker；
3. 解析 PDF；
4. 创建当前页面附近的 Canvas；
5. 加载该 PDF 的增量批注记录。

批注模式失败时保留原生快速阅读，并显示错误信息。

### PDF.js 缓存

PDF.js 和 Worker 包含在项目中，但不在 Service Worker 安装阶段预下载。第一次打开应用不会额外下载约 1.7 MB 的批注引擎。第一次真正进入批注模式后，浏览器再缓存它们。

## 七、保存状态

页面明确显示：

- `待保存`
- `正在保存`
- `已保存`
- `保存失败`

笔迹增量写入完成后立即反馈。结构元数据采用防抖保存。页面进入后台或关闭时，会等待当前增量写入并执行兜底保存。

## 九、仍然存在的边界

1. 真实 Apple Pencil 并发输入必须在 iPad 上验证。
2. 浏览器原生 PDF 能否嵌入右侧面板由浏览器决定；已保留“系统打开”兜底。
3. PDF 批注模式一定比原生快速阅读更重，因为需要 PDF.js 栅格化。
4. 图片仍采用旧版内联 Data URL；大量高清图片会成为下一阶段存储瓶颈。
5. 完整 JSON 备份大型 PDF 仍会慢，因为主动备份必须把 Blob 编码进迁移文件。
6. 单文件前端无法达到 Notability 的原生 PDFKit、Metal/Core Graphics 和系统 Pencil 管线性能。

## 十、建议验收顺序

1. 连续书写 30 秒，观察是否仍有 4–5 秒周期冻结。
2. 每写完一笔立即用手指移动画布。
3. 导入 100 KB、5 MB、50 MB 三类 PDF。
4. 只使用“快速阅读”，确认显示速度。
5. 再主动进入“批注模式”，确认首屏和批注保存。
6. 关闭应用后重新打开，检查笔迹、PDF 和批注恢复。
7. 执行完整备份，再在另一浏览器导入。
